聚合搜索平台

2023/10/11 项目

记录一些puff-Search的笔记

目前的问题：前端目前只能一次搜索出所有数据，无法单独获取某一类数据。接入新的数据源时，都需要在前端增加对应的 axios 调用，后端同事也要增加 service，controller，以及在 facade 中需要增加许多控制调用的代码，重复代码多，系统十分冗余。

解决方案：

  1. 新增一个type字段：
  • 前端传入type
    • 如果type为空，搜索出所有数据
    • 如果type不为空：
      • 如果type合法，查出对应数据
      • 如果type不合法，报错

  1. 为了更轻松地接入更多数据源，考虑使用适配器模式

    适配器模式

    1. 定制统一的数据源接入规范（标准接口）：
    • 什么数据允许接入？
    • 数据源接入时需要满足什么要求？
    1. 当数据源已经支持了搜索，但是原有方法参数和我们规范不一致时，使用适配器模式：通过适配器转换，让两个系统完成对接。

系统中使用到的其他设计模式：

门面模式

在原本系统中，AggregateSearchController类中的代码抽象出来一个新的类facade类，该类就是门面类（controller中不宜堆积太多代码），只需要在controller中调用抽象出来的facade类中的方法即可。（原本AggregateSearchController类实现的就是门面类的功能，但是没有将其中代码抽象成一个类）

当调用你系统（接口）的客户端觉得麻烦的时候，就应该思考，是不是可以抽象一个门面了

注册器模式

提前通过一个map或者其他类型存储好后面需要调用的对象。

效果：替代了if…else…，代码量大幅度减少，可维护可扩展

高亮显示

String contentHighLight = postIdContentHighLight.get(postId);
if (StrUtil.isNotBlank(contentHighLight)) {
    post.setContent(contentHighLight);
}

在高亮显示时能直接替换掉原有content是因为原本需要高亮展示的就不是完整的正文内容，而是只需要匹配到searchText的关系字段（只需要部分内容就好）

Java操作Es的几种方式

  • 使用ES提供的java操作ES的java-api-client 官网

  • 使用ES提供的highLevelRestClient

  • 使用Spring Data Rlasticsearch

    • 第一种方式: 使用继承ElasticsearchRepository<PostEsDTO, Long>，默认提供了简单的增删改查，多用于可预期的、相对没那么复杂的查询、自定义查询，返回结果相对简单直接。

    • 第二种方式: Spring 默认给我们提供的操作es的客户端对象 ，ElasticsearchRestTemplate，也提供了增删改查，它的增删改查更灵活，适用于更复杂的操作，返回结果更完整，但需要自己解析。对于复杂的查询，建议用第二种方式。
      三个步骤:
      1．取参数

      2．把参数组合为ES支持的搜索条件

      3.从返回值中取结果

一些报错

  • elasticSearch启动报错

    [ERROR][o.e.i.g.GeoIpDownloader  ] [06A8DD4C-A847-4] exception during geoip databases update
org.elasticsearch.ElasticsearchException: not all primary shards of [.geoip_databases] index are active

    因为他启动时会去更新地图的一些数据库，这里直接禁掉即可，用到时再说，在配置文件conf/elasticsearch.yml，增加配置

    ingest.geoip.downloader.enabled: false
