InnoDB的MVCC

2024/2/23 Mysql 高并发分布式

Mysql的InnoDB通过MVVC+锁实现了事务。其中MVVC解决隔离性问题，具体而言，MVVC就是为了实现读写并发冲突不加锁。这个读指的是快照读（副本），而非当前读。

什么是MVVC

MVVC（Multi version Concurrency Control）即多版本并发控制，用于提高数据库的并发性能，用更好的方式处理读-写冲突。使得在读写冲突时，能做到不加锁，非阻塞并发读。

当前读

像 select ... lock in share mode(乐观锁)、select ... for update(悲观锁)、update、insert、delete这些操作都是当前读，这些操作读取的都是记录的最新版本，同时会对记录加锁，保证其他并发事务不能修改当前记录。

快照读

不加锁的select操作就是快照读，即不加锁的非阻塞读（事务级别不能是串行化）。快照读的实现基于MVVC（可以认为MVVC是行锁的变种），但它在很多情况下避免了加锁操作，降低了加锁开销。但是因为它读到的可能不是最新版本。（快照读会读取事务开始时数据库中的数据版本，而不受其他事务的影响）

MVCC和快照读的关系

  • MVVC实际是 “保持数据的多个版本来避免读写冲突”（根据隔离级别判断读取那个版本的数据）
  • 快照读会读取事务开始时数据库中的数据版本，这就是MVCC的一种应用（实现在高并发环境下提供一致性视图的读取操作）

MVVC的作用

三种并发场景

  • 读并发：不存在线程安全问题
  • 读写并发： 可能遇到脏读、幻读、不可重复读等问题
  • 写并发：脏写、写倾斜（分析不了一点）

MVVC的作用

在并发读写数据库时，可以做到在读操作时不用阻塞写操作，写操作也不用阻塞读操作，提高了数据库并发读写的性能 同时还可以解决脏读，幻读，不可重复读等事务隔离问题，但不能解决更新丢失问题。

  • MVCC + 悲观锁 MVCC解决读写冲突，悲观锁解决写写冲突
  • MVCC + 乐观锁 MVCC解决读写冲突，乐观锁解决写写冲突

这种组合的方式就可以最大程度的提高数据库并发性能，并解决读写冲突，和写写冲突导致的问题

MVVC的实现原理

MVVC的实现主要通过记录中的***四个隐式字段、undo log、redo view(读视图)***来实现的。

隐式字段

  • DB_ROW_ID 6byte, 隐含的自增ID（隐藏主键），如果数据表没有主键，InnoDB会自动以DB_ROW_ID产生一个聚簇索引
  • DB_TRX_ID 6byte, 最近修改(修改/插入)事务ID：记录创建这条记录/最后一次修改该记录的事务ID
  • DB_ROLL_PTR 7byte, 回滚指针，指向这条记录的上一个版本（存储于rollback segment里）（配合undo log）
  • DELETED_BIT 1byte, 记录被更新或删除并不代表真的删除，而是删除flag变了

undo~

在MySQL中，Read View（读视图）是用于实现多版本并发控制（MVCC）的关键组件之一。Read View主要用于确定在给定事务开始时应该看到哪些数据版本，以确保事务读取到一致的数据视图。

当一个事务开始时，它会创建一个Read View，其中包含一个系统版本号（system version number）。这个系统版本号是一个递增的全局计数器，每次有事务修改数据时就会递增。

当一个事务需要读取数据时，它会使用当前系统版本号来确定应该读取哪个数据版本。具体步骤如下：

事务开始时，获取当前系统版本号作为Read View的版本号。
当事务需要读取数据时，会检查数据行的DB_TRX_ID（事务ID）和DB_ROLL_PTR（回滚指针）。
通过DB_TRX_ID和DB_ROLL_PTR可以找到对应数据行的最新版本，然后检查该版本的事务ID是否小于等于Read View的版本号。
如果是，则表示该数据版本是可见的，事务可以读取该数据；如果不是，则表示该数据版本对当前事务不可见，需要忽略。
通过这种方式，Read View可以根据事务开始时的系统版本号和数据行的事务ID来确定对事务可见的数据版本，从而实现了一致性读取。
